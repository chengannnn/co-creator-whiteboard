## Codebase Patterns
- Typecheck command: `npm run typecheck` (runs tsc on both backend and frontend)
- Corner modifier enablement logic is computed in App.tsx and passed as `isCornerModifierEnabled` prop to UnifiedToolbar (not computed locally in UnifiedToolbar)
- `handleRoundCornerToggle` in App.tsx already handles both drawing-time (Condition A) and attribute-modifier (Condition B) scenarios — it filters selectedIds for rectangle/rhombus and applies borderRadius
- `isSharpCornerEnabled` is derived from `!isRoundCornerEnabled` when enabled; no separate useState

## [2026-04-16] - US-001 (v2.1.7)
- Implemented isCornerModifierEnabled with OR logic for corner modifier buttons
- Renamed `canRoundCorner` to `isCornerModifierEnabled` and moved it from a local variable in UnifiedToolbar to a prop passed from App.tsx
- Condition A: activeTool is rectangle/rectangle-solid/rhombus/rhombus-solid (existing logic)
- Condition B: activeTool === 'select' AND selectedIds.length > 0 AND at least one selected element is rectangle/rhombus
- Button enabled when Condition A OR Condition B is true
- Updated isSharpCornerEnabled derivation in App.tsx to use the same OR logic so Sharp Corner button is also enabled during selection
- handleRoundCornerToggle already correctly handles Condition B (filters eligible elements, applies borderRadius, pushes history, calls onSceneMutate)
- Files changed:
  - `packages/frontend/src/components/UnifiedToolbar.tsx` (renamed canRoundCorner to isCornerModifierEnabled prop, updated all button references)
  - `packages/frontend/src/App.tsx` (added isCornerModifierEnabled computation, updated isSharpCornerEnabled derivation)
- **Learnings for future iterations:**
  - Button enablement logic lives in App.tsx (centralized), not in UnifiedToolbar — this ensures both parent state and child props stay in sync
  - The existing `handleRoundCornerToggle` works for both Condition A and Condition B since it already filters selectedIds for eligible element types
  - Typecheck command: `npm run typecheck`
